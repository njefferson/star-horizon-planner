# Plan & direction review — 2026-07-17

An outside review of NOTES.md (the plan) and v1 as built (18 commits, build-order
steps 1–7 complete on `main`). Verified claims carry links. Corrections that are
plain fact are applied directly to NOTES.md in the same commit as this file;
everything else is a recommendation and stays here until accepted.

## Verdict

The plan is strong and v1 executed it faithfully. All seven steps landed in
order with step-labeled commits; 56/56 headless tests pass in ~0.7 s with zero
dependencies, and the tests are real — independent spherical-astronomy
identities (Polaris ≈ latitude, −0.833° sunset), the 350°→0° azimuth seam,
zenith dead-zone interval splitting, EQ-mode relaxation, legacy migration,
backup round-trips. The instrument-agnostic FOV bet already paid off: S50 and
S30 are computed from specs, never hardcoded, and ZWO's published 2.46°
diagonal FOV for the S30 validates the computed 2.13° × 1.20°. No stubs, no
TODOs, no debug leftovers.

Three factual errors in the plan, one process gap, and a set of v1 loose ends —
together they reorder what to do next.

## Corrections (verified)

### 1. The novelty claim was overstated — and that raises the capture feature's priority
The novelty map said a custom horizon has "no good free tool." Telescopius
(free) supports per-location custom horizon profiles with CSV az/alt import:
<https://forum.telescopius.com/t/request-video-tutorial-on-custom-horizon-scanning-and-import/1182>.
The genuine moat is narrower:

- the **phone-sensor capture UX** (walk the yard, sweep the treeline) — no free
  tool does this;
- **offline-first PWA** (Telescopius needs a connection);
- the **zenith dead-zone "second horizon"** for alt-az smart scopes — genuinely
  novel;
- **horizon-aware polar alignment**.

Consequence: v1 as shipped — a manual 36-row editor — is roughly what
Telescopius already gives away. **Sensor-trace capture isn't a roadmap
nice-to-have; it is the product.** The About dialog repeated the overstated
claim and needs the same correction.

### 2. `webkitCompassHeading` returns MAGNETIC north, not true north
Apple's documentation:
<https://developer.apple.com/documentation/webkitjs/deviceorientationevent/1804777-webkitcompassheading>.
The plan's sensor-capture item said "webkitCompassHeading for true-north
azimuth" — as written, captured horizons would be rotated by the local magnetic
declination, up to ~±15° across the US, i.e. more than one full 10° azimuth
bin. Fix: make **calibration against the Sun's computed azimuth the primary
flow** (one sighting corrects declination AND local magnetic interference;
astronomy-engine already computes the Sun's azimuth), with a bundled
declination model as fallback. Android needs `deviceorientationabsolute`
handling as well.

### 3. The S30's sensor is the Sony IMX662, not the IMX462
ZWO's specs: <https://www.seestar.com/blogs/faq/s30>. `src/data/instruments.js`
claimed both scopes share the IMX462. Same 1920×1080 @ 2.9 µm geometry, so
every computed number was already right — but the repo culture is "compute, do
NOT guess." (Fixed in this commit.)

## Process gap

NOTES.md mandates "ship each step to `staging` for the on-device pass before
merging to `main`," and both workflows wire a `staging` branch — but the branch
was never created; all 18 commits went straight to `main`, i.e. production. The
NEEDS-HIS-HANDS v1 checks (PWA install, offline behaviour, iPad drag feel, the
no-sites first-run journey) have no recorded evidence of having happened.
Likewise, the headless-Chromium UI tests the plan's own verification section
calls for don't exist — all ~1,375 lines of `src/ui/` were untested. The model
layer's excellent coverage makes the asymmetry stark: the two riskiest UI
pieces (horizon-editor drag, night-graph render/scrub) are exactly what the
plan said to test.

**From this review on: work lands on `staging`, gets the on-device pass, then
merges to `main`.**

## v1 loose ends (code survey)

- **Custom-instrument UI missing** — the model layer fully supports custom
  scopes (add/remove/persist/backup-bundled) but Settings only showed the two
  presets; the "plan ANY telescope" promise had no UI.
- **CC-BY-SA attribution not user-visible** — OpenNGC credit existed in
  data/build files but the About dialog showed none; CC-BY-SA-4.0 requires
  visible attribution.
- **Stale copy** — About said "early scaffold"; Settings footer said
  sites/export "arrive in later steps" (they shipped).
- **Vestigial bird-app code** — `ui/panzoom.js` (231 lines, imported nowhere,
  still precached by the SW; the plan meant it for the night-graph scrub but
  the graph hand-rolled its own), `dom.js` `scoreScale()`/`sparkline()`,
  `model/location.js` superseded by `sites.js` except one geolocation helper.
- **NOTES.md said `s50.*` localStorage keys** — the implementation (correctly)
  standardized on `horizon.*`.
- **Horizon model flexibility** — profiles are fixed 36-bin; Stellarium import
  resamples (lossy); altitudes clamp to [0, 90] so a below-zero horizon
  (hilltop, balcony) can't be represented. Fine for manual editing — but decide
  BEFORE sensor capture ships whether stored profiles allow arbitrary azimuth
  points; sensor sweeps produce denser data than 10° bins, and real treelines
  have gaps narrower than 10°.
- Nitpick — `visibility.js` interval edges use bracketing-sample *midpoints*
  while the comment said "linear-interpolated crossing" (±1 min at the 2-min
  step; comment fixed to match the code).
- **Missing from the plan entirely: Moon interference.** Angular separation
  from the target + illumination is a classic planning factor every serious
  tool shows; `moonInfo` already existed in `astro.js`, so it's cheap and
  high-value on the visibility table.

## Recommended direction

1. **v1.1 polish** — custom-instrument editor UI; visible attribution; stale
   copy; prune dead code; `navigator.storage.persist()` + backup nudge (a
   measured horizon is precious data living in evictable browser storage);
   Moon-interference indicator. Plus: create `staging` and actually run the
   on-device pass v1 skipped.
2. **Sensor-trace horizon capture** — the moat, now demonstrably so.
   Sun-calibration primary; settle horizon storage resolution first.
3. **Polar-align, split in two** — the horizon-aware "can I even see the pole
   from this site" check is pure computation over existing data and can ship
   nearly free; the live "point to the pole" aid comes after capture, since it
   shares the compass-calibration stack.
4. **Weather overlay** (Open-Meteo) — the highest recurring nightly value.
5. Thumbnails + framing overlay → map-pin terrain horizon → sky segmentation
   (stretch), as already planned.
